# 简历检查 MVP 方案

Date: 2026-04-08
Status: Draft

## 1. 目标校正

当前第一目标不是做展示型个人网站，而是先做一个本地运行的「简历检查 MVP」。

这个 MVP 的核心任务是：
- 用户上传简历文档
- 系统完成文本提取
- 基于 AIPM 规则输出评审结果
- 给出可直接替换的改写建议

这一步优先解决“简历评审与改写”，而不是一次性覆盖 AIPM 全流程。

## 2. 当前知识库的作用

当前导入的知识库承担三层角色，但优先级如下：

1. 作为简历检查 MVP 的评审规则来源和内容素材来源
2. 作为后续 AIPM 全流程文档检查的规则底座与语料库
3. 在后续阶段，才作为网站展示与内容生产素材库

结论：
这批知识首先服务“检查”，不是先服务“展示”。

## 3. MVP 核心定义

### 3.1 主入口

MVP 主入口应为「上传文档」，不是手动粘贴大段文本。

推荐上传方式：
- 点击上传
- 拖拽上传

MVP 首批支持格式：
- `.pdf`
- `.docx`

可选内部测试格式：
- `.txt`
- `.md`

但在正式用户界面里，重点应表达为“上传简历文档”，而不是“复制文本进去”。

### 3.2 MVP 解决的问题

用户上传一份简历后，系统需要回答 3 个问题：

1. 这份简历当前质量怎么样？
2. 主要问题出在哪里？
3. 哪些内容应该怎么改？

### 3.3 MVP 最小闭环

完整闭环应为：

1. 上传简历文档
2. 提取文本
3. 执行简历评审
4. 输出评分与问题列表
5. 输出改写建议

只要这 5 步跑通，MVP 就成立。

## 4. 页面结构

### 页面 1：上传页

作用：
- 上传简历文件
- 展示文件解析状态
- 用户确认开始评审

必须有：
- 文件上传区域
- 支持拖拽上传
- 支持重新上传
- 展示文件名、大小、类型
- “开始评审”按钮

MVP 不强调：
- 多文件同时上传
- 复杂历史记录
- 账号体系

### 页面 2：评审结果页

作用：
- 展示当前简历的整体质量与问题分布

必须有：
- 总分（100 分）
- 分角色评分
  - HR
  - PM
  - 技术
  - 设计
  - 项目管理
- 问题列表
  - `P0` 严重问题
  - `P1` 重要问题
  - `P2` 优化建议
- 每条问题的原因说明

建议增加：
- 命中规则说明
- 原文片段引用

### 页面 3：改写结果页

作用：
- 给出用户可直接替换进简历的文案

必须有：
- 针对问题的逐条改写建议
- 优化后的项目表达示例
- 优化后的完整版本草稿

建议增加：
- 原文 / 改写后 对照展示

## 5. MVP 评审内容结构

MVP 的评审引擎先不要追求“最智能”，而要追求“稳定可解释”。

第一版评审至少覆盖下面几类规则：

### 5.1 结构完整性
- 项目是否表达完整：背景 -> 做了什么 -> 结果
- 是否只有职责，没有结果
- 是否只有工具，没有方案

### 5.2 表达质量
- 是否出现“工具等于能力”的表达
- 是否出现英文术语加括号解释
- 是否存在套话、空话、弱表达
- 是否多个项目句式重复

### 5.3 证据充分性
- 优势是否有经历支撑
- 数据是否可解释
- 结果是否可量化

### 5.4 AI PM 相关性
- 是否体现模型选型能力
- 是否体现用户需求拆解
- 是否体现技术边界理解
- 是否体现数据评估能力
- 是否说明为什么使用 AI

## 6. 数据来源与规则来源

MVP 的规则来源分两层：

### 第一层：显式规则
来自：
- `docs/KNOWLEDGE_BASE.md`
- `AGENTS.md`
- `PRODUCT_BRIEF.md`
- `MVP_SCOPE.md`

这部分负责定义可直接执行的判断框架。

### 第二层：素材与案例语料
来自导入的飞书知识库：
- 求职
- 新奥集团
- 个人成长
- 子博设计
- 写作文章

这部分负责：
- 提供高质量表达参考
- 提供 AI PM 项目包装样例
- 提供改写语气与结构参考

## 7. 建议的技术边界

为了满足“本地运行 + MVP + 不做复杂后端”，建议这样约束：

- 前端框架：Next.js
- 样式：Tailwind CSS
- 运行方式：本地
- 文件解析：在 Next.js 服务端处理
- 暂不引入数据库
- 暂不做登录
- 暂不做云部署

### 上传解析建议

MVP 推荐优先支持：
- `pdf`
- `docx`

并明确以下限制：
- 扫描版 PDF 暂不保证可用
- 图片 OCR 暂不进入 MVP
- 多文件合并评审暂不进入 MVP

## 8. MVP 分级

### P0：先跑起来
- 初始化 Next.js 项目
- 完成上传页、结果页、改写页
- 支持上传 `pdf/docx`
- 成功解析文件文本
- 先用假数据评审结果跑通完整链路

### P1：结果可用
- 把知识库硬规则接入评审逻辑
- 输出真实问题列表
- 输出基础改写建议
- 让结果具备可解释性

### P2：体验优化
- 更好的结果展示
- 更强的原文/改写对比
- 更稳的解析与异常处理
- 更完整的评语组织方式

## 9. 明确不做

当前 MVP 不做：
- 手动大段粘贴文本作为主入口
- 登录系统
- 用户管理
- 数据库
- 云部署
- 多文档批量检查
- OCR
- AIPM 全流程文档检查完整能力

注意：
“AIPM 全流程文档检查”不是不做，而是放在简历检查 MVP 之后。

## 10. 与后续 AIPM 全流程的衔接

当前 MVP 是第一阶段，后续自然扩展路径如下：

### 第二阶段
- 项目经历检查
- 项目封装检查
- 面试表达生成

### 第三阶段
- PRD / BRD / MRD / MVP / 技术方案等文档检查
- 面向 AIPM 的全流程文档质量评审

因此，当前 MVP 不应写死成“只会检查简历”的架构，而应保留一个更通用的概念：

- `document upload`
- `document parse`
- `review pipeline`
- `rewrite pipeline`

只是当前第一种文档类型是“简历”。

## 11. 我建议采用的 MVP 口径

一句话版本：

> 先做一个本地运行的 AIPM 简历检查工具，用户通过上传 PDF / DOCX 简历文件获取评分、问题列表和改写建议；它既是 MVP，也为后续全流程文档检查打基础。

## 12. 当前最合理的下一步

按开发顺序，下一步应该是：

1. 初始化 Next.js 项目
2. 定义上传文件与评审结果的数据结构
3. 先完成上传 -> 假结果 -> 改写结果的页面打通
4. 再逐步接入真实评审规则
